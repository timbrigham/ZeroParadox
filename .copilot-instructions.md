# Copilot Instructions for README.md Maintenance

This file documents the workflow and best practices for maintaining the Zero Paradox README.md file. Follow these guidelines when updating the main README.

## Document Structure

The README should maintain this hierarchical order:

1. **Title and Date** - `# The Zero Paradox - Project Index`
2. **"What This Is"** - High-level introduction to the framework
3. **"The Central Result"** - The core theorem and derivation chain
4. **"What This Is Not"** - Clarifications on what the framework does NOT claim
5. **"Document Index"** - Tables of all available documents
6. **"Axiomatic Commitments"** - Formal commitments and principles
7. **"Status of All Major Open Questions"** - Tracked open items (if any remain)
8. **"Reading Order"** - Paths for different reader types (with clickable links)
9. **"Notes on Development"** - Credits and contributor information
10. **"Repository and Version History"** - Git/versioning guidance + link to historical/ folder
11. **"Purpose of This Repository"** - Why this repo exists
12. **"License"** - Licensing information with link to LICENSE file
13. **"Citation"** - Proper citation format
14. **"Contact"** - How to reach out for collaboration

## Formatting Standards

### File Links
- **Display text**: Use clean names without file extensions or version numbers
  - ✓ `[ZP-A Lattice Algebra](ZP-A_Lattice_Algebra_v1_1.pdf)`
  - ✗ `[ZP-A Lattice Algebra v1.1.pdf](...)`
  - ✗ `[ZP-A_Lattice_Algebra_v1_1.pdf](...)`

- **Link targets**: Point to the latest version (without `-1`, `-2` suffixes)

### Text Formatting
- Use **regular hyphens** (`-`) throughout, not em dashes (`—`)
  - Exception: Mathematical operators (→ arrows) are acceptable
  
### Tables
- Maintain consistent column alignment
- Use meaningful column headers (File, Document, Version, Contents)
- Remove redundant information between display text and columns
  - Version numbers go in the Version column, not in the display name

## Document Index Best Practices

### Entry Point Table
- Always include README.md and Zero_Paradox_Foreword.pdf
- Remove broken links immediately
- Test all links for validity before committing

### Formal Ontology Documents
- List all 7 documents (ZP-A through ZP-H)
- Remove `.docx` alternatives if they don't exist in repo
- Include version numbers in the Version column only
- Each entry: `[Display Name](actual_filename.pdf)` where Display Name has no version

### Illustrated Companion Documents
- Keep all 6 companions in alphabetical order
- These provide accessible entry points for general readers
- Link in Reading Order should point to this section with anchor tag

### Supporting Documents
- Include `ZP_Tools_and_Methods.pdf` (and any others that exist)
- Remove broken links to files that don't exist (e.g., build_*.py, PDF_Rendering_Standards.md)

## Historical Files Management

### Moving Old Versions
- Move old versions with suffixes (`-1`, `-2`, etc.) to the `historical/` folder
- Keep the top-level directory clean and focused on current versions
- This makes development history visible without Git access
- Maintain a `historical/README.md` explaining what's there

### Version Numbers in Filenames
- Current versions have format: `FILENAME_v#_#.pdf` (no suffix)
- Old versions have format: `FILENAME_v#_#-N.pdf` (with numeric suffix)
- Always link to the non-suffixed version in the main README

## Reading Order Section

### Structure
Include distinct paths for different audiences:
1. **General reader** - Foreword → Illustrated Companion → ZP-E Companion
2. **Mathematician** - Formal path through ZP-A through ZP-E
3. **Category theory extension** - ZP-G and ZP-H (post-ZP-E)
4. **Process/methods** - ZP Tools and Methods

### Link Implementation
- **All documents** should be clickable links, not plain text
- Links should point directly to the files for easy access
- For "any Illustrated Companion": Use anchor link to `#illustrated-companion-documents-general-reader` section to show all options

**Format**: `[Display Name](filename.pdf)` not `filename.pdf`

## Validation Checklist

Before finalizing README updates:

- [ ] No file extensions in display text (except where necessary)
- [ ] No version numbers in display text (use Version column instead)
- [ ] All em dashes replaced with regular hyphens
- [ ] Reading Order has clickable links for all documents

## Common Updates

### Adding a New Document
1. Add to appropriate section (Formal Ontology or Supporting)
2. Use clean display name without extension or version
3. Link to the newest version (no `-1`, `-2` suffixes)
4. Add Version column entry
5. Test the link

### Removing Broken Links
- Check repo contents: `ls *.pdf`
- Verify file exists before linking
- Remove entries that point to missing files
- Consider: Is this a file that should be created, or genuinely not part of this branch?

### Moving Version Files to Historical
1. Identify files with numeric suffixes (`-1`, `-2`, etc.)
2. Create `historical/` folder if it doesn't exist
3. Move old version files: `mv filename-*.pdf historical/`
4. Update main README links to point only to current versions
5. Add note in "Repository and Version History" section linking to historical folder
6. **Update `historical/README.md` table** (see below)

### Maintaining the Historical Folder

When moving files to the `historical/` folder, update the `historical/README.md` file with a table entry:

**Table format:**
```
| File | Date Moved | Description |
|------|-----------|-------------|
| [Display Name](filename-suffix.pdf) | YYYY-MM-DD | Description of what this version represents |
```

**Guidelines:**
- **File column**: Preserve the actual archived filename in the link text, including affixes and suffixes
  - ✓ `[ZP-A_Lattice_Algebra_v1_1-1.pdf](ZP-A_Lattice_Algebra_v1_1-1.pdf)`
  - ✓ `[Zero_Paradox_Foreword-2.pdf](Zero_Paradox_Foreword-2.pdf)`
  - ✗ `[ZP-A Lattice Algebra](ZP-A_Lattice_Algebra_v1_1-1.pdf)`
- **Date Moved**: Use YYYY-MM-DD format (current date when file is moved)
- **Description**: Brief note about what iteration this was or why it was superseded
  - Example: "Earlier iteration of ZP-C (v1.3)"
  - Example: "Previous version before reorganization"
- Keep entries in chronological order (oldest first or newest first consistently)

### Updating Open Questions Status
- Maintain the "Status of All Major Open Questions" table
- Update entries as theorems are proven or questions resolved
- Format: `| Item | Status |` with descriptive status text
- Reference specific theorems where applicable (e.g., "ZP-E T5")

## Notes

- The README is the entry point; it should guide readers to the right resources
